# 維運總手冊

## 1. 這個專案在做什麼

這個專案把景氣循環判讀做成可重複更新的網站。

核心目標只有三件事：

1. 每月抓台灣與美國的相關資料
2. 根據固定規則整理成網站可讀的結果
3. 把結果部署到正式網站

網站不是即時系統，也不是前端直接去抓外部資料。
正確理解方式是：

1. 用 Python 抓資料
2. 算出網站總判讀和三個鏡頭
3. 產出 JSON 和靜態頁面
4. 部署到 Cloudflare Pages

## 2. 專案定位

這是一個「每月重跑一次」的靜態網站流程，不是常駐後端。

正式分工如下：

- GitHub Actions：負責每月自動更新
- Python 程式：負責抓資料、整理資料、產出網站內容
- `dist/`：最後可部署的網站成品
- Cloudflare Pages：負責正式網站

不建議另外開同名 Worker 當正式站入口，避免混淆。

## 3. 重要目錄

- `src/market_phase_detector/`
  核心 Python 程式
- `frontend/src/`
  靜態頁面、樣式、前端互動
- `data/`
  最新資料、歷史資料、網站文案資料
- `dist/`
  最後可部署的靜態網站
- `tests/`
  測試
- `wrangler.toml`
  Cloudflare Pages 設定

## 4. 主要程式分工

### `src/market_phase_detector/cli.py`

整個專案的總入口，負責：

- 建立最新資料
- 建立歷史資料
- 建立網站文案資料
- 產出 `dist/`

### `src/market_phase_detector/live_pipeline.py`

把外部資料整理成專案內部要用的觀測值。

### `src/market_phase_detector/collectors/`

負責抓資料。

- `us_fred.py`：抓美國 FRED
- `tw_official.py`：抓台灣國發會
- `tw_external.py`：抓台灣其他補充資料

### `src/market_phase_detector/engine/`

負責網站總判讀規則。

- `tw_rules.py`：台灣總判讀
- `us_rules.py`：美國總判讀
- `state_machine.py`：把候選結果轉成較穩定的正式相位

### `src/market_phase_detector/lenses/`

負責三個鏡頭各自的判讀。

- `izaax.py`
- `urakami.py`
- `marks.py`

### `src/market_phase_detector/exporters/json_exporter.py`

把結果寫進 `data/*.json`。

### `src/market_phase_detector/site_builder.py`

把前端和資料組成 `dist/`。

### `frontend/src/app.js`

前端互動總入口，讀 JSON 後組出首頁和國別頁。

### `frontend/src/styles.css`

網站樣式。

## 5. 資料來源與判讀方向

### 美國

目前主要用 FRED。

已使用的重點資料包括：

- 製造業工業生產
- 初領失業救濟金
- 薩姆規則
- 殖利率曲線
- 高收益債利差

### 台灣

目前主要用國發會景氣資料，加上一些外部補充資料。

已使用的重點資料包括：

- 景氣對策信號
- 領先與同時指標
- 出口
- 勞動壓力相關資料
- 部分市場與利率資料

## 6. 網站產出流程

高層流程如下：

1. 抓外部資料
2. 整理成專案內部資料
3. 算網站總判讀
4. 算三個鏡頭
5. 寫出 `data/latest.json`
6. 寫出 `data/history/*.json`
7. 寫出 `data/site-content.json`
8. 把前端和資料組成 `dist/`
9. 部署到 Cloudflare Pages

## 7. 本地重建

先跑：

```bash
pytest -q
python -m market_phase_detector.cli
```

如果要本地看頁面：

```bash
cd dist
python -m http.server 8765
```

打開：

- `http://localhost:8765/`
- `http://localhost:8765/tw/`
- `http://localhost:8765/us/`

## 8. Cloudflare 設定

建議維持以下做法：

- 保留一個 Pages 專案：`market-phase-detector`
- 正式網址：`https://market-phase-detector.pages.dev/`
- 可保留分支預覽網址，但不要把它當正式站
- 不使用同名 Worker 當正式網站入口

## 9. 自動更新與手動重發

### 自動更新

GitHub Actions 每月排程執行整套流程：

1. 抓資料
2. 整理資料
3. 產出 `dist`
4. 部署到 Cloudflare Pages

### 手動重發

手動重發分兩種：

- GitHub Actions 手動執行：重跑整套更新流程
- Cloudflare Pages 後台重新部署：只重發最近一次成功的網站成品

這個專案建議：

- 要重抓資料，用 GitHub Actions 手動執行
- 要快速把同一版網站再送一次，用 Cloudflare Pages 重新部署

## 10. 上線步驟

如果要手動部署：

```bash
npx wrangler pages deploy dist --project-name market-phase-detector
```

上線前至少確認：

1. 測試有過
2. `data/latest.json` 有更新
3. `data/history/index.json` 有更新
4. `data/site-content.json` 有更新
5. `dist/` 已重建
6. 本地首頁、台灣頁、美國頁都看過

## 11. 常見檢查順序

### 如果頁面內容怪怪的

先看這個順序：

1. `data/latest.json`
2. `data/history/index.json`
3. `data/site-content.json`
4. `dist/`
5. Cloudflare Pages 線上頁面

### 如果畫面不對，但資料看起來對

優先檢查：

1. `dist/` 是否真的是最新版本
2. 你打開的是不是正式網址
3. 瀏覽器是不是還吃到舊快取
4. 樣式 class 有沒有跟別的區塊重名互撞

### 如果月份不對或跳號

優先檢查：

1. `data/history/index.json`
2. 各月份歷史檔
3. 月份對齊邏輯

## 12. 不要做的事

- 不要讓前端直接連外部資料源
- 不要把網站總判讀寫成某位作者的結論
- 不要手改歷史索引檔來補資料
- 不要只改前端文字卻不重建資料檔
- 不要把 Pages、Worker、預覽網址混成同一件事

## 13. 已知限制

- 有些外部來源穩定度一般，重跑時可能受對方站點影響
- 這個網站是月頻，不是日頻或即時更新
- 某些台灣資料有時格式會變，需要同步調整解析規則
